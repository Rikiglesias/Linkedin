Questo file contiene la checklist dei miglioramenti ancora sensati e realmente prioritari per portare la codebase a uno stato più robusto, coerente e manutenibile, evitando rewrite impulsive o task già coperti in modo sostanziale.

## 1. Frontend, UX Operativa e Surface Prodotto (Priorità: ALTA - CRITICO)
Il frontend attuale ha già molte capacità operative reali, ma va reso più modulare, coerente e difendibile nel tempo.

Il file contiene SSE/WebSocket connection management, polling loop, voice recognition, UI event bindings, rendering orchestration e localStorage prefs — tutto nello stesso file. Attualmente `public/style.css` (20KB) usa valori hardcoded per colori, spacing e font. ## 2. Architettura Backend, Runtime Boundaries e Scalabilità (Priorità: ALTA)
Il punto critico non è “fare microservizi” in astratto, ma separare bene responsabilità, lifecycle e failure domain.

- [ ] **Separare il processo API/Dashboard dal processo Worker/Browser** con entrypoint, healthcheck e lifecycle indipendenti. Attualmente `src/index.ts` (616 righe) gestisce sia il comando `dashboard` (che avvia Express su porta 3000) sia tutti i comandi operativi (`run`, `autopilot`, `run-loop`) nello stesso processo. Il problema: un crash del browser Playwright abbatte anche l'API dashboard, e viceversa un memory leak dell'API blocca i job. Approccio: creare `src/entrypoints/api.ts` e `src/entrypoints/worker.ts`, ciascuno con il proprio graceful shutdown. In `docker-compose.yml` diventano due servizi (`bot-api` e `bot-worker`) con healthcheck indipendenti. Il DB (SQLite o PG) è già condiviso via file/connessione — non serve IPC. La coda job (`jobs` table) funge da broker naturale.
- [ ] **Formalizzare contratti tra orchestrator, queue, worker e API**. Attualmente i 16 worker in `src/workers/` hanno firme eterogenee: `processInviteJob(payload, context)`, `processInboxJob(payload, context)`, `runFollowUpWorker(context)`, `createAndPublishPost(page, options)`. Creare un'interfaccia `WorkerProcessor<TPayload>` con metodo `process(ctx: WorkerContext, payload: TPayload): Promise<WorkerExecutionResult>`. Il `jobRunner.ts` (732 righe) ha già uno switch/case per tipo job (righe ~100-200) — sostituire con un registry `Map<JobType, WorkerProcessor>`. Lo stato condiviso in-memory da ridurre: `activeBrowsers` Set globale in `launcher.ts`, `proxyFailureUntil` Map in `proxyManager.ts`, `circuitStates` Map in `integrationPolicy.ts` — tutti non persistenti, persi al restart.
- [ ] **Valutare un broker persistente (Redis + BullMQ)** solo come step successivo alla separazione API/worker. Attualmente la coda è una tabella `jobs` in SQLite/PG con colonne `status`, `scheduled_at`, `attempts`, `locked_at`. Il pattern `lockNextQueuedJob()` in `repositories/jobs.ts` usa `UPDATE ... WHERE status='QUEUED' LIMIT 1` come lock ottimistico. Questo funziona per 1 worker ma non scala a N worker paralleli senza race condition (SQLite non ha `FOR UPDATE SKIP LOCKED`; PG sì). Redis + BullMQ risolverebbe: job deduplication nativa, delayed jobs, retry con backoff, dashboard Bull Board. Prerequisito: separazione processi completata (task precedente).
- [ ] **Aggiornare `docker-compose.yml`** per supportare scaling separato. File attuale: `docker-compose.yml` (91 righe) con 3 servizi (db, bot, dashboard-nginx). Modifiche: (a) separare `bot` in `bot-api` e `bot-worker`; (b) il worker non espone porte, l'API espone 3000; (c) nginx fa proxy solo verso `bot-api`; (d) aggiungere `restart: unless-stopped` + `deploy.resources.limits.memory: 2G` per il worker (Playwright è memory-hungry); (e) aggiungere volume named per `data/session*` condiviso tra API e worker se necessario.
- [ ] **Ridurre config drift con profili ambiente**. `src/config/domains.ts` (418 righe) costruisce la config leggendo ~200 variabili `.env` senza grouping logico. `src/config/validation.ts` (398 righe) ha ~50 regole di validazione ma solo sui valori individuali, non sulle combinazioni cross-dominio. Miglioramenti: (a) introdurre profili `CONFIG_PROFILE=dev|staging|production` con override di default per ambiente; (b) aggiungere validazioni cross-dominio (es. `COMPLIANCE_ENFORCED=true` dovrebbe forzare `HARD_INVITE_CAP <= COMPLIANCE_MAX_HARD_INVITE_CAP`); (c) raggruppare le ~200 variabili in sezioni con prefix coerenti nel `.env.example` (già fatto parzialmente, ma mancano i gruppi per proxy, timing, growth model).
- [ ] **Splittare `src/api/server.ts` (61KB, 1603 righe)**: god file monolite. Estrarre in moduli: `middleware/auth.ts`, `middleware/security.ts`, `routes/controls.ts`, `routes/leads.ts`, `routes/stats.ts`, `routes/security.ts`, `routes/ai.ts`, `routes/sync.ts`. Il file server diventa solo bootstrap e composizione.
- [ ] **Splittare `src/salesnav/bulkSaveOrchestrator.ts` (112KB)**: file più grande della codebase. Decomporre in: `pageNavigator.ts`, `leadExtractor.ts`, `saveProcessor.ts`, `retryHandler.ts`, `orchestrator.ts`.
- [ ] **Migrazione ESM**: `package.json` usa `"type": "commonjs"` con target TS `ES2022`. Frizione con librerie ESM-only e futuro Node. Pianificare migrazione a `"type": "module"` con aggiornamento import paths.
- [ ] **Repository pattern incompleto**: i repository in `src/core/repositories/` mescolano query SQL raw con business logic. Separare nettamente data access (SQL/CRUD) dai service che contengono logica.
- [ ] **Error handling utility uniforme**: mix inconsistente di `catch {}`, `catch(() => {})`, `.catch(() => null)`. Creare utility `safeAsync(fn, fallback?)` che almeno logga a livello debug. Zero errori silenziati senza traccia.
- [ ] **Dependency Injection leggera**: il codice usa import diretti ovunque, rendendo difficile mockare e testare. Introdurre un `AppContext` passato esplicitamente con database, config, logger, proxyManager.
- [ ] **SQL normalization cache LRU**: `PostgresManager.normalizeSql()` esegue 8+ regex replace ad ogni query. Le stesse SQL string vengono ripetute migliaia di volte. Cacheare il risultato con LRU map.

## 3. Anti-Ban, Browser Reliability e Self-Healing (Priorità: MEDIA)
La direzione corretta è aumentare coerenza e verificabilità, non spingere su tecniche sempre più evasive e fragili.

- [ ] **Rafforzare self-healing DOM-first**. `src/browser/uiFallback.ts` (505 righe) ha già un sistema sofisticato: ranking candidati con `scoreSelectorCandidate()` (stabilità, confidenza, success history), context cache LRU (300 entry, 6h TTL), dynamic selectors dal DB. Aree da migliorare: (a) **Shadow DOM coverage** — `page.locator()` non penetra Shadow DOM chiusi; LinkedIn usa Web Components in alcune sezioni (messaging, notifiche). Aggiungere `page.locator('pierce/...')` o `page.evaluate()` con `querySelector` ricorsivo su `shadowRoot`; (b) **verifica post-azione** — dopo ogni click/type, verificare che l'azione abbia avuto effetto (es. dopo click "Connect", verificare che il modale nota sia apparso); (c) `src/selectors/learner.ts` (13KB) gestisce il ranking persistente — aggiungere metriche di "selector drift" per rilevare quando LinkedIn cambia i class name (es. `artdeco-button` → nuovo prefisso).
- [ ] **Vision fallback controllato**. `src/salesnav/visionNavigator.ts` (12KB) e `src/salesnav/computerUse.ts` (17KB) usano il VisionProvider (GPT-5.4 → Ollama fallback) per click coordinate-based. `src/workers/hygieneWorker.ts` (righe 66-75) lo usa come fallback quando i CSS selectors falliscono sul bottone "Pending/Withdraw". Regola: usarlo **solo** come ultimo resort dopo 2+ fallimenti CSS, mai come path primario. Aggiungere un flag `visionFallbackUsed: boolean` nel `WorkerExecutionResult` per tracciare quanto spesso viene invocato e verificare che non stia mascherando selettori rotti.
- [ ] **Misurare incoerenze fingerprint/stealth**. I 19 moduli stealth in `stealthScripts.ts` (634 righe) patchano separatamente: WebRTC, webdriver, plugins, languages, chrome mock, permissions, headless guards, hwConcurrency, battery, notification, audio, deviceMemory, colorDepth, storage seeder, OS consistency, language consistency, CDP leaks, WebGL renderer, iframe chrome. Il rischio: una patch è coerente internamente ma incoerente con un'altra (es. `deviceMemory=8` ma UA dice "Android mobile" dove 4GB è più realistico). Approccio: creare un test `fingerprint-coherence.test.ts` che verifica le combinazioni mobile/desktop e OS-specific. Confrontare i valori patchati con dataset reali (es. FingerprintJS open dataset).
- [ ] **Regressioni testabili anti-detection**. Nessun test attuale verifica i claim stealth. Creare `src/tests/stealth.vitest.ts` che: (a) esegue `buildStealthInitScript()` e verifica che l'output JS sia parsabile; (b) lancia un browser headless con lo script iniettato e verifica `navigator.webdriver === false`, `navigator.plugins.length >= 3`, `window.chrome.runtime` esiste, `Notification.permission === 'default'`; (c) confronta `navigator.hardwareConcurrency` con il valore atteso per il fingerprint. Usare `playwright` in test mode (già disponibile come dependency).
- [ ] **Criteri rollback/disable per stealth sperimentali**. Attualmente il flag `STEALTH_SCRIPTS_SKIP_IF_CLOAK` in `.env` permette di saltare sezioni quando CloakBrowser è attivo, ma non c'è un meccanismo per disabilitare singole sezioni stealth in base al loro impatto. Aggiungere un `.env` flag `STEALTH_SKIP_SECTIONS=storage_seeder,battery` (CSV) che bypassa sezioni specifiche senza bisogno di CloakBrowser. In `stealthScripts.ts` il meccanismo `_skip` Set (riga 71) è già pronto — basta passare il CSV parsato.
- [ ] **Fingerprint deterministico per account**: attualmente il fingerprint (UA, viewport, ecc.) viene scelto random dal pool ad ogni lancio. LinkedIn correla che lo stesso account ha impronte diverse ogni sessione. Fix: usare `sha256(accountId + weekNumber)` come seed per selezionare deterministicamente dal pool. Cambiare solo ogni 1-2 settimane (simula aggiornamento browser).
- [ ] **Canvas noise con PRNG**: il noise in `launcher.ts` usa pattern fisso `pixelIndex % 2/3/5` per i segni — deterministico e rilevabile statisticamente. Fix: usare PRNG con seed dal fingerprint ID per noise pseudo-casuale ma deterministico per sessione.
- [ ] **WebGL renderer pool da espandere**: solo 2 valori possibili (`Apple GPU` / `ANGLE Intel Iris Xe`). LinkedIn può profilare che il 100% delle sessioni usa lo stesso renderer. Fix: pool di 10-15 renderer realistici (Intel UHD, AMD Radeon, NVIDIA GeForce, vari ANGLE) selezionati per fingerprint.
- [ ] **humanWindDown prevedibile**: alla chiusura il bot naviga **sempre** verso `/feed`. Pattern costante. Fix: variare destinazione (feed, notifiche, homepage, /mynetwork, oppure restare sulla pagina corrente) con distribuzione pesata.
- [ ] **JA3 fingerprint pool (solo se CycleTLS attivo)**: il JA3 nel `.env` è usato SOLO da CycleTLS (`USE_JA3_PROXY=true`). Senza CycleTLS, Playwright usa lo stack TLS nativo di Chromium e la stringa JA3 è ignorata. Se CycleTLS è attivo: pool di 5-10 JA3 reali per evitare fingerprint singolo. Se CycleTLS è spento (default) con proxy mobile Oxylabs: il proxy carrier fornisce già un TLS handshake nativo — non serve JA3 spoofing. **Deprioritizzare** per volumi bassi (< 30 inviti/giorno). Il costo di setup (proxy MITM, certificati CA, processo separato) non giustifica il beneficio a basso volume.
- [ ] **Typing speed variabile per contesto**: `humanType` usa velocità fisse. Un umano digita più veloce su testi brevi e più piano su testi lunghi/complessi (numeri, maiuscole, punteggiatura). Modulare WPM in base a lunghezza e complessità del testo.
- [ ] **Session cookie anomaly detection**: `sessionCookieMonitor.ts` traccia la maturity ma non rileva anomalie come cookie invalidato prematuramente o cambio improvviso di session token (segnale di ban imminente). Aggiungere alert specifico.

## 4. Sicurezza, Governance e Threat Model Reale (Priorità: ALTA)
Serve blindare meglio le superfici davvero esposte e riallineare ciò che è dichiarato in documentazione con ciò che succede davvero nel codice.

- [ ] **Allineare documentazione e implementazione reale**. `SECURITY.md` (riga 146) dichiara "Dashboard frontend is served from local compiled assets with strict CSP (`script-src 'self'`)" ma `index.html` carica Chart.js da CDN esterno (violazione). `SECURITY.md` (riga 1-2) dichiara "No stealth/evasion browser plugins" ma `stealthScripts.ts` inietta 19 sezioni di evasion. Allineare la documentazione alla realtà o viceversa. Elenco discrepanze da risolvere: CSP vs CDN, stealth claim, export endpoints non documentati in SECURITY.md, dashboard session TTL (12h nel codice, non documentato).
- [ ] **Ridurre auth bootstrap via query string**. `src/frontend/apiClient.ts` (righe 66-91) implementa `bootstrapSessionFromUrl()`: legge `?api_key=...` dall'URL, crea una session cookie, poi rimuove il parametro dall'URL con `history.replaceState`. Il problema: l'API key transita nell'URL (browser history, server logs, referer header). Fix: sostituire con un form POST di login nella dashboard (username/password → session cookie) oppure limitare il bootstrap URL a localhost-only. Se mantienuto, aggiungere un warning in SECURITY.md.
- [ ] **Integrare un Secret Manager Esterno** per produzione. Attualmente TUTTI i segreti (OpenAI key, Supabase service role key, proxy password, Telegram bot token, dashboard API key, webhook secret, Apollo API key) sono nel `.env` da 18KB in chiaro. Per produzione: integrare con **Doppler**, **Infisical** (self-hosted, gratuito), o **1Password CLI** (`op inject`). L'integrazione minima è un wrapper in `config/env.ts` che prova prima il secret manager e poi fallback su `.env`. In alternativa: Docker Secrets (`/run/secrets/`) per i deployment containerizzati.
- [ ] **Implementare 2FA/TOTP per dashboard**. L'auth attuale (`server.ts` righe 462-577) supporta: API key, Basic auth, session cookie, trusted IP. Nessun secondo fattore. Per aggiungere TOTP: (a) generare un TOTP secret per utente (libreria `otpauth` o `speakeasy`); (b) aggiungere un endpoint `POST /api/auth/totp/verify` che valida il codice 6 cifre; (c) il session cookie viene emesso solo dopo API key/Basic + TOTP valido; (d) salvare il TOTP secret cifrato nel DB o in `.env` (`DASHBOARD_TOTP_SECRET`). Alternativa più semplice per single-user: TOTP via Telegram bot (manda il codice su Telegram, l'utente lo inserisce).
- [ ] **Risk engine: livello analitico spiegabile**. `src/risk/riskEngine.ts` (279 righe) usa un modello rule-based con pesi fissi: `errorRate * 40 + selectorFailureRate * 20 + pendingRatio * 25 + challengeCount * 10 + inviteVelocityRatio * 15`. I pesi sono hardcoded e non calibrati su dati reali. Aggiungere: (a) un endpoint `/api/risk/explain` che mostra il contributo di ogni fattore allo score corrente (trasparenza); (b) un log strutturato per ogni decisione STOP/WARN/LOW_ACTIVITY con i valori che l'hanno causata; (c) un meccanismo di "weight tuning" basato su historical outcomes (quante volte un WARN si è poi tradotto in un challenge reale). Non sostituire con ML opaco — il modello rule-based è più sicuro per il caso d'uso.
- [ ] **Checklist periodica SECURITY.md / THREAT_MODEL.md**. Aggiungere un check automatico nel `security-advisor` (`src/core/securityAdvisor.ts`, 564 righe) che verifica: (a) la data di modifica di `SECURITY.md` e `THREAT_MODEL.md` non supera `SECURITY_ADVISOR_DOC_MAX_AGE_DAYS` (config esistente); (b) i controlli dichiarati nei doc corrispondono alla config runtime (es. se SECURITY.md dice "2FA required" ma `DASHBOARD_TOTP_SECRET` non è configurato → warning). Il security advisor esegue già periodicamente in `run-loop` — aggiungere questi check alla lista in `runSecurityAdvisor()`.
- [ ] **Docker container gira come root**: il `Dockerfile` non ha `USER` directive. Aggiungere `RUN groupadd -r botuser && useradd -r -g botuser botuser` + `USER botuser` prima del `CMD`. Riduce drasticamente la superficie d'attacco in caso di RCE.
- [ ] **nginx senza HTTPS/TLS**: `nginx.conf` serve solo HTTP porta 80, nessuna configurazione SSL. In produzione: aggiungere TLS con cert Let's Encrypt (o documentare esplicitamente che HTTPS è delegato a reverse proxy esterno tipo Cloudflare/Caddy). Aggiungere `Strict-Transport-Security` header.
- [ ] **PG backup non implementato**: `backupDatabase()` per PostgreSQL ritorna solo una stringa informativa. Implementare `pg_dump` via child_process oppure integrare con backup managed (Supabase, pg_dump cron in sidecar Docker).
- [ ] **VACUUM INTO con interpolazione SQL**: in `db.ts` il backup SQLite usa `VACUUM INTO '${safePath}'` con path interpolato nella stringa SQL. Nonostante la regex di validazione, è un anti-pattern. Preferire VACUUM senza path + file copy, oppure usare il metodo `.backup()` di better-sqlite3.
- [ ] **Rate limit solo per IP**: il rate limiter Express è basato esclusivamente su IP. Dietro un proxy/NAT, tutti gli utenti condividono il limite. Per gli endpoint autenticati, basare il rate limit anche sul session token o API key.

## 5. Database, Persistenza e Strategia Data Layer (Priorità: MEDIA)
Le decisioni qui vanno prese su profili di carico reali, non per moda architetturale.

- [ ] **Profilare query hot-path e lock contention**. I repository in `src/core/repositories/` (14 file, ~230KB totali) contengono centinaia di query SQL raw. Le hot-path sono: `lockNextQueuedJob()` in `jobs.ts` (chiamata ad ogni job cycle), `getDailyStat()` in `stats.ts` (chiamata ~10 volte per ciclo orchestrator), `getRiskInputs()` in `stats.ts` (3 subquery aggregate). In SQLite, `lockNextQueuedJob` usa `UPDATE ... WHERE id = (SELECT id ... LIMIT 1)` che non ha row-level lock — sotto carico concorrente può causare `SQLITE_BUSY`. Il `busy_timeout` è 5000ms (`db.ts` riga 406). Approccio: aggiungere `EXPLAIN QUERY PLAN` logging per le top-20 query in un mode diagnostico, misurare tempo medio e P99, identificare index mancanti.
- [ ] **Layer type-safe incrementale** per le nuove aree complesse. Il candidato migliore è **Kysely** (zero runtime overhead, query builder type-safe, supporta SQLite + PG dialetti). Drizzle è un'alternativa ma ha un ORM layer che aggiunge complessità. Strategia: (a) introdurre Kysely solo per i nuovi moduli (es. `repositories/campaigns.ts`, `repositories/featureStore.ts` che sono i più recenti); (b) generare i tipi DB con `kysely-codegen` dal schema esistente; (c) mantenere i repository legacy con SQL raw fino a quando non vengono toccati per altre ragioni. Non fare mai una migrazione big-bang — il costo di riscrivere le ~500 query esistenti non giustifica il beneficio.
- [ ] **Partitioning PostgreSQL**. Le tabelle candidate sono: `lead_events` (cresce linearmente con ogni transizione lead), `job_attempts` (ogni tentativo di job), `outbox_events` (ogni evento sincronizzato), `run_logs` (ogni ciclo). In `src/db/migrations/` ci sono 50 migrazioni ma nessuna definisce partitioning. In PG16: usare `PARTITION BY RANGE (created_at)` con partizioni mensili. Prerequisito: essere in produzione su PG (attualmente SQLite in dev). Non implementare su SQLite — non supporta partitioning.

Fix: cacheare un hash delle migrazioni applicate e skippare `applyMigrations()` se il set non è cambiato. Anche `ensureColumn*()` (righe 294-316) fa ~20 ALTER TABLE idempotenti ad ogni boot — servono solo per retrocompatibilità. Dopo che tutte le colonne esistono, queste chiamate sono no-op ma leggono il schema DB ogni volta.
- [ ] **SQL normalization cache** (già in TODO sezione 2). `normalizeSql()` fa 8 regex per ogni query PG. Con ~50 query per ciclo job × 10 job per run = 500 regex chains. Una `Map<string, string>` cache elimina il 95% del lavoro.

### Velocizzare operazioni SU LinkedIn (rischio basso se fatto bene)
- [ ] **Navigazione pagine: `domcontentloaded` vs `load`**. Molti worker usano `waitUntil: 'domcontentloaded'` che è già veloce. Ma alcuni posti usano ancora `waitUntil: 'load'` che aspetta TUTTE le risorse (immagini, tracking pixels). Verificare e uniformare a `domcontentloaded` dove possibile — LinkedIn carica tracking async che non serve per i selettori DOM.
- [ ] **Ridurre screenshot inutili**. `challengeHandler.ts` fa screenshot full-page per analisi vision ad ogni challenge detection. Ma `detectChallenge()` in `auth.ts` è chiamata preventivamente anche quando non c'è challenge (verifica di sicurezza). Aggiungere early-exit: se `detectChallenge()` ritorna `false`, non fare screenshot. Attualmente il screenshot viene fatto DOPO la detection, ma il costo di `page.screenshot()` (~200ms) si accumula.
- [ ] **Parallelizzare operazioni non-browser durante i delay**. Durante `humanDelay()` e `interJobDelay()` (30-300 secondi di attesa) il processo è idle. Usare quel tempo per: (a) pre-fetch del prossimo profilo lead con `page.evaluate(() => fetch(...))` in background (rischioso); oppure più sicuramente (b) eseguire il batch enrichment parallelo, (c) sincronizzare con Supabase/n8n, (d) eseguire cleanup outbox. Attualmente `loopCommand.ts` fa queste operazioni sequenzialmente DOPO il run — farle DURANTE i delay tra job.
- [ ] **Browser GC più aggressivo**. `performBrowserGC()` in `launcher.ts` (righe 515-530) chiude le pagine extra e forza `HeapProfiler.collectGarbage`. Ma viene chiamato solo su richiesta esplicita. Automatizzare: dopo ogni N job (es. 10), eseguire GC. Playwright/Chromium accumula memoria (~50-100MB per pagina visitata senza GC) — dopo 20 profili il browser rallenta visibilmente.

### Anti-ban improvements — velocità come DIFESA
- [ ] **Sessioni più corte = meno esposizione**. Attualmente `INTER_JOB_MIN_DELAY_SEC=120` e `INTER_JOB_MAX_DELAY_SEC=300` nel `.env`. Con 10 inviti al giorno × 120-300s tra job = 20-50 minuti di sessione MINIMA solo per i delay. Il totale con navigazione e typing è ~60-90 minuti. LinkedIn monitora la durata totale della sessione. Un umano reale fa 20 minuti di attività, non 90. **Fix**: ridurre `INTER_JOB_MAX_DELAY_SEC` a 180 (da 300) e compensare con `WARMUP_TWO_SESSIONS_PER_DAY=true` (già attivo): 2 sessioni da 30 min è meglio di 1 sessione da 90 min. Questo è più veloce E più sicuro.
- [ ] **Budget split intelligente tra sessioni**. Con `WARMUP_TWO_SESSIONS_PER_DAY=true`, il budget è diviso 50/50 tra le due sessioni (`getSessionBudgetFactor()` in `sessionWarmer.ts` riga 58-62). Ma la mattina (9-12) ha tipicamente acceptance rate più alto — sarebbe meglio allocare 60% la mattina e 40% il pomeriggio. Rendere il split configurabile: `WARMUP_SESSION1_BUDGET_FACTOR=0.6`.
- [ ] **Skip profili già visitati nel giorno**. Quando il bot naviga su un profilo per invitare/messaggiare, LinkedIn registra una "profile view". Se lo stesso profilo viene visitato 2 volte nello stesso giorno (es. invito fallito + retry), sono 2 profile view per lo stesso target — pattern sospetto. Aggiungere un cache in-memory `visitedProfilesToday: Set<string>` e skippare la navigazione se il profilo è già stato visitato. Il `PROFILE_VIEW_DAILY_CAP=50` nel `.env` controlla il totale ma non i duplicati.
- [ ] **Randomizzare l'ordine di processing dei job**. `lockNextQueuedJob()` processa i job in ordine FIFO (ORDER BY scheduled_at, id). Un pattern costante "prima tutti gli inviti, poi tutti i check, poi tutti i messaggi" è rilevabile. Aggiungere un piccolo shuffle: `ORDER BY scheduled_at + (RANDOM() * 60)` per SQLite o `ORDER BY scheduled_at + (random() * interval '60 seconds')` per PG — crea variazione nell'ordine senza cambiare la priorità temporale.
- [ ] **Adaptive inter-job delay basato su risposta server**. `httpThrottler.ts` (4KB) già traccia i response time delle API LinkedIn. Se LinkedIn risponde più lentamente del solito (P95 > 2x baseline), è un segnale di throttling soft. Integrare: quando il throttler rileva rallentamento, aumentare automaticamente `interJobDelay` del 50% per la sessione corrente. Attualmente `httpThrottler` è read-only — non influenza i delay.

## 14. Strategie Avanzate Cross-Modulo e Anti-Detection Intelligente (Priorità: ALTA)
Collegamenti tra moduli diversi che singolarmente funzionano ma insieme creano pattern rilevabili, più strategie furbe per aumentare l'acceptance rate (meno pending = meno rischio ban).

### Moduli esistenti non collegati tra loro (potenza sprecata)
- [ ] **`httpThrottler` disconnesso dal sistema di delay**. `httpThrottler.ts` calcola `shouldSlow` e `shouldPause` in tempo reale, ma il segnale è letto solo 1 volta in `jobRunner.ts` — e solo per logging. `humanDelay()`, `interJobDelay()`, e il `scheduler` NON consultano il throttler. Collegamento da fare: in `humanBehavior.ts` `interJobDelay()` (riga 611), leggere `session.httpThrottler.getThrottleSignal()` e se `shouldSlow=true` moltiplicare il delay ×1.5; se `shouldPause=true` aggiungere una pausa coffee break forzata di 3-5 minuti. Questo crea un feedback loop reattivo: LinkedIn rallenta → il bot rallenta automaticamente → LinkedIn smette di sospettare.
- [ ] **`weeklyStrategyEnabled=false` nel `.env` — strategia settimanale disabilitata**. `strategyPlanner.ts` ha un piano settimanale raffinato (lunedì: alto inviti, giovedì: focus messaggi, venerdì: wind-down) ma è disattivato. Lo `scheduler.ts` lo chiama ma riceve sempre factor 1.0. **Abilitare** `WEEKLY_STRATEGY_ENABLED=true` — è gratis, zero rischio, e rende il pattern settimanale più umano. LinkedIn correla: un bot fa la stessa cosa ogni giorno; un umano ha ritmi diversi per giorno della settimana.
- [ ] **`sessionMemory.pacingFactor` calcolato ma poco sfruttato**. `sessionMemory.ts` calcola un `pacingFactor` (0.5-1.2) basato sulla storia recente: se i giorni passati hanno avuto challenge, il pacing scende; se sono stati tranquilli, sale. Lo `scheduler.ts` lo usa per modulare il budget. Ma `humanBehavior.ts` NON lo usa — i delay intra-azione restano identici indipendentemente dalla "tensione" recente. Collegamento: passare il `pacingFactor` a `humanDelay()` e `interJobDelay()` come moltiplicatore. Dopo un giorno con challenge → il bot il giorno dopo è naturalmente più lento. Dopo 3 giorni tranquilli → può essere leggermente più veloce.
- [ ] **`timingOptimizer` e `abBandit` non comunicano**. Il timing optimizer (`timingOptimizer.ts`, 498 righe) decide QUANDO inviare l'invito (quale ora/giorno). L'AB bandit (`abBandit.ts`, 504 righe) decide COME inviare (quale variante di messaggio). Ma non si incrociano: lo stesso messaggio viene inviato alla stessa ora indipendentemente dalla variante. Collegamento avanzato: certe varianti di messaggio funzionano meglio a certe ore (es. messaggi formali la mattina, informali il pomeriggio). Cross-segmentare timing × variante nell'AB test.

### Anti-detection: pattern cross-sessione che LinkedIn rileva
- [ ] **Login hour costante = pattern bot**. Il bot lancia la sessione sempre alla stessa ora (quando l'utente esegue il comando o quando PM2 parte). Un umano varia: oggi entra alle 9:15, domani alle 9:47, dopodomani alle 10:03. `sessionMemory.ts` traccia `login_hour` ma nessuno usa il dato per VARIARE l'orario di inizio. Fix: in `loopCommand.ts`, prima di iniziare il ciclo, aggiungere un delay random di 0-45 minuti (`Math.random() * 45 * 60 * 1000`) per sfasare l'orario di login tra sessioni. Costo: qualche minuto di attesa. Beneficio: LinkedIn non vede login alle 09:00:00 ogni giorno.
- [ ] **Durata sessione costante = pattern bot**. Se il bot processa 10 inviti con delay 120-300s ogni volta, la durata sessione è sempre ~30-50 minuti. Un umano a volte sta 15 minuti, a volte 45. Fix: dopo aver calcolato il budget giornaliero, applicare un fattore random giornaliero (0.7-1.3) alla durata totale. Alcuni giorni il bot fa 7 inviti in 20 minuti, altri ne fa 13 in 50 minuti. La media resta uguale ma la varianza è umana.
- [ ] **Ratio inviti/messaggi/check costante tra giorni**. Il bot fa sempre X inviti, Y check, Z messaggi nello stesso ordine e proporzione. Un umano ha giorni dove si concentra sugli inviti e giorni dove risponde ai messaggi. Il `weeklyStrategyPlanner` (task sopra) risolve parzialmente questo, ma manca una componente random: aggiungere un "mood factor" giornaliero che sbilancia il budget ±20% random tra invite e message. Es. oggi il bot è "in mood inviti" (budget invite ×1.2, message ×0.8), domani il contrario.
- [ ] **Pagine visitate sempre nello stesso ordine**. I worker processano: profilo → click connect → (opzionale) add note → send. Sempre la stessa sequenza. Un umano a volte: guarda il profilo, scrolla l'experience, torna su, legge un post, poi decide di connettersi. In `inviteWorker.ts` aggiungere una probabilità (20%) di navigare a `/recent-activity/` del target PRIMA di tornare al profilo e cliccare Connect. LinkedIn vede una visita più organica e meno "arriva → clicca → se ne va".

### Strategie furbe per massimizzare acceptance rate (meno pending = meno ban)
- [ ] **"Warm touch" prima dell'invito**. LinkedIn traccia le interazioni pre-invito. Se PRIMA di invitare qualcuno il bot: (a) visualizza il profilo del target 1-2 giorni prima (profile view), (b) mette Like a un suo post, (c) segue il target — l'acceptance rate sale del 30-50% perché il target vede "chi ha visitato il tuo profilo" e riconosce il nome quando arriva l'invito. Il sistema drip campaign (`campaignEngine.ts`) supporta già step `VIEW_PROFILE`, `LIKE_POST`, `FOLLOW` — ma il workflow default `send-invites` NON li usa. Creare un workflow "warm-touch → wait 1-2 days → invite" come template consigliato.
- [ ] **Like strategici sui post del target (non random)**. `organicContent.ts` mette Like su post random nel feed. Ma un Like su un post del target specifico che stai per invitare è molto più efficace di un Like random. Modificare: in `interactionWorker.ts` quando `actionType='LIKE_POST'`, navigare al profilo del lead target → activity → Like il post più recente. Il target vede la notifica "X ha reagito al tuo post" e poi riceve l'invito da X — coincidenza apparente che aumenta la fiducia.
- [ ] **Inviti con nota personalizzata basata su contesto reale**. L'acceptance rate con nota personalizzata è 30-40% vs 15-20% senza nota. Ma `INVITE_WITH_NOTE=false` nel `.env` attuale — un'opportunità persa. Con `AI_PERSONALIZATION_ENABLED=true` (già attivo) e `INVITE_NOTE_MODE=ai`, il bot genera note personalizzate via GPT-5.4. Abilitare `INVITE_WITH_NOTE=true` e `INVITE_NOTE_MODE=ai`. Costo: ~$0.002 per nota. Beneficio: raddoppio del tasso di accettazione → dimezzamento del pending ratio → dimezzamento del rischio ban.
- [ ] **Targeting intelligente: invitare prima chi è più probabile che accetti**. Il `leadScorer.ts` (3KB) assegna uno score ai lead ma il `scheduler.ts` non ordina i job per score — processa in ordine FIFO. Fix: quando il scheduler crea i job, ordinare i lead per `lead_score DESC` (i più promettenti prima). Se il budget giornaliero è 10 inviti, è meglio invitare i 10 con score più alto che i 10 più vecchi. Meno inviti rifiutati = pending ratio più basso = meno rischio.
- [ ] **Ritirare inviti pending vecchi PRIMA di invitarne di nuovi**. Il `hygieneWorker.ts` ritira inviti pending dopo `PENDING_INVITE_MAX_DAYS=30` giorni. Ma 30 giorni di inviti pending si accumulano e gonfiano il pending ratio. Fix furbo: abbassare `PENDING_INVITE_MAX_DAYS` a 14-21 giorni. Chi non accetta in 2 settimane probabilmente non accetterà mai. Ritirandoli prima: (a) il pending ratio scende più velocemente; (b) il risk engine è più rilassato; (c) LinkedIn vede un utente che "pulisce" le richieste — comportamento umano normale.

### Ottimizzazioni di intelligence operativa
- [ ] **Auto-pausa prima dei maintenance window di LinkedIn**. LinkedIn fa maintenance notturno e deploy infrastrutturali (tipicamente martedì/mercoledì notte US Pacific = mercoledì/giovedì mattina presto EU). Durante questi periodi i selettori possono cambiare e le API sono instabili. Aggiungere nel `loopCommand.ts` un check: se l'ora locale è tra 03:00-06:00, saltare il ciclo con un log "maintenance_window_skip". Evita errori inutili e selector failure che alzano il risk score.
- [ ] **Correlazione cross-account**. Con `MULTI_ACCOUNT_ENABLED`, due account sullo stesso proxy Oxylabs o sulla stessa macchina creano una correlazione. LinkedIn può vedere che due account fanno login dallo stesso IP nello stesso minuto. Fix: sfasare le sessioni di almeno 2-3 ore tra account. Se account1 lavora 9-11, account2 lavora 14-16. Mai sovrapposizione. Verificare in `loopCommand.ts` che i due account non siano mai in sessione contemporanea.
- [ ] **Tracciare le "impressioni" per calcolare il ROI reale**. Attualmente il bot traccia inviti, accettazioni, messaggi, risposte. Ma non traccia: (a) quanti profili sono stati visualizzati senza azione (browsing organico); (b) quanti Like sono stati dati; (c) quanti post sono stati creati e visti. Queste metriche servono per calcolare il "funnel conversion rate" completo e capire se le azioni decoy hanno un effetto positivo sull'acceptance. Aggiungere contatori in `daily_stats`: `profile_views_organic`, `likes_given`, `posts_created_views`.

### DA RIPENSARE — Approccio sbagliato da rifare diversamente

- [ ] **RIPENSARE: Doppia patch WebGL (launcher.ts + stealthScripts.ts sezione 18)**. Non basta "unificarle" — il problema è più profondo. `launcher.ts` inietta noise Canvas/WebGL come `addInitScript` prima della pagina, poi `stealthScripts.ts` patcha `WebGLRenderingContext.getParameter` come secondo `addInitScript`. L'ordine di esecuzione dipende dall'ordine di `addInitScript` — non deterministico tra versioni Playwright. **Ripensare**: un UNICO modulo `src/fingerprint/stealthPatches.ts` che genera TUTTO lo script da iniettare (canvas noise + WebGL renderer + WebGL vendor) in un singolo `addInitScript`. La selezione del renderer viene dal fingerprint pool — nessun hardcoding nei script. `launcher.ts` chiama `buildUnifiedStealthScript(fingerprint)` e inietta una sola volta.
- [ ] **RIPENSARE: Il dual-DB adapter pattern (SQLiteManager / PostgresManager)**. L'idea era buona (supportare entrambi i DB) ma l'implementazione via `normalizeSql()` con 8 regex è fragile. Ogni nuova query che usa una funzione SQLite-specifica (es. `json_set`, `GROUP_CONCAT`, `SUBSTR`) richiede una nuova regex. E non tutte le conversioni sono testabili — `json_set` in SQLite vs `jsonb_set` in PG hanno semantiche diverse sui NULL. **Ripensare**: scegliere UN database target per la produzione (PG) e mantenere SQLite SOLO per dev/test con un subset di funzionalità. Oppure usare Kysely che genera SQL dialect-aware e rende il problema inesistente. L'attuale approccio "scrivi SQLite, converto on-the-fly" è un debito tecnico che cresce con ogni migrazione.
- [ ] **RIPENSARE: God files come architettura accettata**. `server.ts` (1603 righe), `bulkSaveOrchestrator.ts` (112KB), `adminCommands.ts` (43KB), `loopCommand.ts` (32KB), `utilCommands.ts` (34KB), `salesNavCommands.ts` (30KB). Sono 6 file che totalizzano ~300KB. Il pattern è sempre lo stesso: un file inizia piccolo, cresce incrementalmente, e nessuno lo splitta perché "funziona". Il problema non è solo la leggibilità — è che un bug in un file da 1600 righe è 10x più difficile da isolare, testare e revieware. **Regola**: nessun file `.ts` dovrebbe superare le 500 righe. Quando raggiunge 400, è il momento di estrarre.
