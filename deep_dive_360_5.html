100% del valore del sistema dipende da LinkedIn. Se LinkedIn cambia la UI radicalmente, blocca il JA3, o implementa detection avanzata, il bot diventa inutile. Nessuna diversificazione (email outreach, altre piattaforme). Nessun "graceful degradation" — o funziona tutto o niente.
MED11 fire-and-forget promises che perdono dati criticiIntero progetto
11 promise lanciate senza await in contesti dove il risultato è importante: runtime flags, session patterns, enrichment results. Se falliscono silenziosamente, il dato è perso. Esempio: recordSessionPattern(...).catch(() => null) — se fallisce, la session memory non impara da quella sessione.
MEDNo incident classification: non distingue "nostro bug" vs "LinkedIn changed"risk/incidentManager.ts
Quando un selettore fallisce, il sistema non sa se è un nostro bug (es. typo nel selettore) o se LinkedIn ha cambiato la UI. Entrambi generano lo stesso tipo di errore. Risultato: l'utente apre un ticket per "il bot non funziona" e servono 30 minuti di debug per capire se il problema è nostro o di LinkedIn.
MEDNo client-facing story: metriche esistono ma non in formato leggibiletelemetry/ + api/
Il sistema traccia KPI dettagliati (acceptance rate, response rate, daily stats). Ma l'unico modo per vederli è via API JSON o CLI. Nessun report settimanale leggibile dal cliente. Il daily report via Telegram è tecnico. Un CEO non vuole vedere {"invites_sent": 28, "acceptance_rate_pct": 34.2}.
